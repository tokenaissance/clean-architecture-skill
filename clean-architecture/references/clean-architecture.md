# 整洁架构完整原则 — Robert C. Martin《架构整洁之道》

> 一句话总结：将系统的核心业务逻辑（策略）与实现细节（框架、数据库、UI）分离，通过依赖反转让所有依赖指向内层高层策略，构建独立于框架、可测试、可替换、易维护的系统。

## 1. 为什么架构重要

**好架构的价值**：大幅节省人力成本，让变更简单快速，缺陷极少，用最小成本最大化功能性和灵活性。

**坏架构的代价**：组件耦合紧密，再小的改动都需要数周恶战；团队士气低落、客户痛苦。

**设计与架构没有本质区别**。架构是高层结构，设计是底层细节，但两者是连续体，共同决定了系统的形态。衡量架构好坏的标准只有一个：**构建和维护系统所需的人力成本**。如果每个版本的人力成本持续上升，架构就失败了。

**两个价值维度**：

| 维度 | 含义 | 紧急性 |
|------|------|--------|
| **行为价值** | 软件按需求正确运行（功能） | 总是紧急的 |
| **架构价值** | 软件易于修改以适应新需求（结构） | 不紧急但极重要 |

大多数团队只关注行为价值而忽视架构价值——这是一个致命错误。不紧急但重要的事（架构）必须优先于紧急但不重要的事（某个具体功能需求）。架构师的职责就是守卫这个优先级。

## 2. 编程范式与约束

三大编程范式不是赋予我们新能力，而是**约束**我们的行为：

| 范式 | 约束 | 架构意义 |
|------|------|---------|
| **结构化编程** | 限制 goto（约束直接转移控制） | 让我们能用分解法拆分模块，用反证法证伪测试 |
| **面向对象编程** | 限制函数指针（约束间接转移控制） | 通过多态实现**依赖反转**——这是 OOP 对架构最重要的贡献 |
| **函数式编程** | 限制赋值（约束可变状态） | 消除竞争条件、并发问题、死锁——不可变性是架构稳定性的基础 |

**OOP 的核心价值不是封装、不是继承，是多态带来的依赖反��能力**。在多态出现之前，源代码依赖的方向只能跟随控制流方向。多态让我们可以在任意位置反转依赖方向——这就是整洁架构的基石。

**函数式编程的启示**：如果资源无限，所有数据都可以不可变。现实中，采用"事件溯源"（Event Sourcing）——只存储事务（增删改的指令），不存储状态，通过回放事务得到状态。用足够的存储和处理能力换取不可变性。

## 3. SOLID 设计原则

SOLID 是中层设计原则，适用于模块和类的组织，目标是创建能容忍变化、易于理解、可被复用的模块结构。

### SRP — 单一职责原则

**常见误解**：每个模块只做一件事——**错！**

**正确定义**：任何一个软件模块都应该只对某一类行为者（actor）负责。

**经典反例**：`Employee` 类同时包含 `calculatePay()`（CFO团队负责）、`reportHours()`（COO团队负责）、`save()`（CTO团队负责）。当 CFO 要求修改计算逻辑时，可能无意破坏 COO 依赖的报表功能。

**解决方案**：
- 将数据与函数分离——三个类共享 `EmployeeData`，各自只包含相关函数
- 或使用 Facade 模式委派调用

**在架构层面**：SRP 表现为共同闭包原则（CCP）和变更轴心（Axis of Change），用于确定架构边界的位置。

### OCP — 开闭原则

**定义**：设计良好的软件应该易于扩展，同时抗拒修改。

这是研究软件架构的**根本目的**。如果小小的需求变化需要大幅修改代码，架构就失败了。

**实现方法**：
1. 用 SRP 拆分职责
2. 用 DIP 组织依赖方向
3. 建立保护层级：高阶组件（业务逻辑）不受低阶组件（UI、数据库）修改影响

**保护层级**：Interactor（业务逻辑）> Controller > Presenter > View。高层不知道低层的存在。

### LSP — 里氏替换原则

**定义**：子类型必须可以替换基类型，不破坏调用者的期望。

LSP 不仅适用于继承，更适用于接口和约定。任何依赖一个接口的代码，都应该能安全地使用该接口的任何实现。

**经典违反**：正方形继承长方形。设置宽度时自动修改高度，破坏了调用者对"长方形"行为的期望。

**在架构层面**：LSP 扩展为接口和协议的互换性。如果一个出租车调度系统依赖 URI 来调度不同公司的车辆，而某个公司的 REST 接口不遵守约定（参数名不同），就需要 if-else 特殊处理——这就是架构层面的 LSP 违反。

### ISP — 接口隔离原则

**定义**：不要让客户端依赖它不使用的方法。

**本质**：依赖了不需要的东西，就会受到不需要的变更的影响。

**在架构层面**：系统 S 依赖框架 F，F 依赖数据库 D。如果 D 有个无关变化，可能迫使 F 重新部署，进而迫使 S 重新部署。不需要的传递性依赖是危险的。

### DIP — 依赖反转原则

**定义**：最灵活的系统是源代码依赖关系只引用抽象类型，而非具体实现的系统。

**例外**：对稳定的具体实现（如标准库的 String 类）无需应用 DIP。**重点关注那些经常变动的具体实现模块。**

**四条编码守则**：
1. 多使用抽象接口，避免使用多变的具体实现类
2. 不要在具体实现类上创建衍生类（继承是最强的依赖）
3. 不要覆盖包含具体实现的函数
4. 避免在代码中写入任何具体实现相关的名字

**工厂模式**：通过 `ServiceFactory` 接口创建具体实例，所有跨越架构边界的依赖都是单向的——具体实现依赖抽象。**控制流方向与源代码依赖方向相反**——这就是"依赖反转"的由来。

## 4. 组件构建原则

### 组件聚合 — 什么应该放在一起？

| 原则 | 含义 | 驱动力 |
|------|------|--------|
| **REP（复用/发布等同原则）** | 复用的粒度等于发布的粒度 | 便于复用 |
| **CCP（共同闭包原则）** | 因同一原因同一时间变化的类放在同一组件 | 便于维护（SRP 的组件版） |
| **CRP（共同复用原则）** | 不要强迫用户依赖他们不使用的东西 | 避免不必要的发布（ISP 的组件版） |

这三个原则形成张力三角：
- REP + CCP 倾向于让组件��大（包含更多东西便于复用和维护）
- CRP 倾向于让组件变小（只包含一起使用的东西）
- 项目早期更偏向 CCP（快速开发），成熟期更偏向 REP（便于复用）

### 组件耦合 — 组件间的关系

**无依赖环原则（ADP）**：组件依赖图必须是有向无环图（DAG）。

打破循环依赖的方法：
1. 应用 DIP，引入接口反转依赖方向
2. 创建新组件，把共享的类提取出去

**稳定依赖原则（SDP）**：依赖方向必须指向更稳定的方向。

- **稳定性**不是指"不变"，而是指"难以改变"——被很多组件依赖的组件是稳定的
- 不稳定的组件（容易变）应该依赖稳定的组件（难以变），反之不行
- 稳定性指标 I = Fan-out / (Fan-in + Fan-out)，I=0 最稳定，I=1 最不稳定

**稳定抽象原则（SAP）**：稳定的组件应该是抽象的。

- 稳定组件必须开放扩展（否则无法修改），通过抽象类和接口实现
- 抽象度 A = 抽象类数 / 总类数
- **主序列**：理想组件落在 (I=0, A=1) 到 (I=1, A=0) 的对角线上
- **痛苦区**：I=0, A=0（稳定且具体，如数据库 schema，改不动）
- **无用区**：I=1, A=1（不稳定且抽象，没人用）

## 5. 软件架构核心思想

### 架构师必须是程序员

不亲身承受设计带来的痛苦，就会迷失方向。架构师必须一直写代码。

### 架构的目标

便于**开发、部署、运行、维护**。其中维护成本最高——主要消耗在"探秘"（寻找修改位置）和"风险"（修改引入缺陷）。

### 核心策略：推迟决策

> **一个优秀的软件架构师应该致力于最大化可选项数量。**

所有软件系统 = 策略（业务规则） + 细节（I/O、数据库、框架）。架构师的目标是以策略为核心，让细节与策略脱钩。

具体建议：
- 早期不必选择数据库系统
- 早期不必选择 Web 服务器
- 早期不必采用 REST
- 早期不必采用依赖注入框架
- **即使公司已指定某技术，好架构师也应假装决策未定**

### 独立性

**康威定律**：系统结构映射组织沟通结构。

**解耦的三个层面**：
1. 按层解耦（UI / 业务逻辑 / 数据库）— 水平分层
2. 按用例解耦 — 垂直切分
3. 解耦模式：源码层次（单体）→ 部署层次（jar/DLL）→ 服务层次（微服务）

**表面性重复 vs 真正重复**：两段看起来相同的代码，如果变更原因和速率不同，不是真正的重复，不要合并！

**保留可选项**：先在单体中做好解耦，需要时再拆分为服务。好架构允许双向演进。

### 策略与层次

**层次**：距离输入/输出越远，层次越高。直接管理 I/O 的策略层次最低。

**规则**：源码依赖与数据流方向脱钩，而与层次挂钩——低层组件依赖高层组件。低层组件是高层组件的"插件"。

## 6. 整洁架构模型

整合六边形架构、DCI、BCE 的统一模型。

### 四层同心圆

| 层（由内到外） | 内容 | 说明 |
|--------------|------|------|
| **业务实体 Entities** | 关键业务规则 + 关键业务数据 | 无论系统存在与否都会存在的规则（如银行利息计算）。最通用、最稳定、最不容易变动 |
| **用例 Use Cases** | 特定应用场景的业务逻辑 | 控制实体间数据流动。输入→处理→输出。不描述 UI 细节 |
| **接口适配器 Adapters** | Controller、Presenter、Gateway | 在内外层数据格式间转换。MVC 框架属于这层 |
| **框架与驱动 Frameworks** | Web 框架、数据库、UI | 所有实现细节，只写胶水代码 |

### 依赖关系规则（The Dependency Rule）

> **源码中的依赖关系必须只指向同心圆的内层，即由低层机制指向高层策略。**

- 内层代码不应引用外层代码中的任何命名
- 外层数据格式不应被内层使用
- 层数可以超过四层，但规则不变

### 跨越边界

用 DIP 解决控制流与依赖方向相反的问题。用例需要调用展示器时，不直接调用（违反规则），而是让用例调用内层接口（输出端口），由展示器实现该接口。利用动态多态反转依赖。

### 跨越边界的数据

使用简单数据结构（DTO、哈希表、函数参数）。**不要传递业务实体对象或数据库记录对象**——采用内层最方便使用的形式。

### 五大特征

1. **独立于框架** — 框架是工具，不是架构
2. **可测试** — 业务逻辑可脱离 UI、数据库、Web 测试
3. **独立于 UI** — UI 可替换不影响业务逻辑
4. **独立于数据库** — 可替换存储技术
5. **独立于外部机构** — 业务逻辑不知道外部接口存在

## 7. 边界的类型与策略

### 边界的存在形式（通信成本递增）

| 形式 | 成本 | 适用场景 |
|------|------|---------|
| **单体内多态** | 最低（函数调用） | 项目早期，需求不确定 |
| **部署组件** (jar/DLL) | 中低（进程内调用） | 团队独立开发、独立部署 |
| **本地进程** | 中（系统调用+上下文切换） | 安全隔离、故障隔离 |
| **服务** | 最高（网络往返，毫秒到秒级） | 物理隔离、独立扩展 |

**所有边界的本质**：管控源码依赖方向，让低层成为高层的插件。

### 不完全边界

完整边界成本高昂。三种折中：

1. **省掉最后一步**：接口和数据结构都设计好，但编译为单一组件。省去多组件管理，但隔离可能随时间弱化
2. **单向边界（策略模式）**：只用一个方向的接口，依赖反转已就位但无反向保护，靠开发者自律
3. **门户模式（Facade）**：连依赖反转都省略，最简单但 Client 传递性依赖所有 Service

### 谦卑对象模式

在每个架构边界处，将难以测试的行为（Humble Object）与容易测试的行为分离。

- **View** 是谦卑对象——只负责填充数据到屏幕，不做处理
- **Presenter** 是可测试对象——格式化数据为字符串/布尔值/枚举值
- **数据库网关**也是谦卑对象边界：接口定义操作，实现类执行 SQL
- ORM 更准确的称呼是"数据映射器"，属于数据库层的谦卑对象

### Main 组件

系统最细节化、最低层次的组件。职责是创建所有工厂类和全局设施，然后将控制权转交最高抽象层。

- Main 是系统中"最脏的组件"
- 依赖注入应在 Main 中完成
- **Main 可视为应用程序的一个插件**——可以为不同环境设计多个 Main

## 8. 实现细节的正确定位

### 数据库只是实现细节

- 数据模型重要，但数据库（工具）不重要
- 不应让数据库行对象在系统内部传递
- 数据库流行的根本原因是磁盘慢需要索引优化，��是技术细节不是架构

### Web 只是实现细节

- Web 只是 I/O 设备之一，是软件行业集中/分布式振荡中的一次浪潮
- 业务规则必须与 UI 解耦
- 架构师应把眼光放长远，不将短期技术振荡放入核心业务逻辑

### 框架只是实现细节

- 使用框架是**单向婚姻**——你承诺了框架，框架没承诺你
- 框架作者解决的是他们自己的问题
- 不要让框架进入内圈，将其作为架构最外圈的细节
- **不要嫁给框架**：如果框架要求基于其基类派生，创建代理类作为插件

### 尖叫的软件架构

系统架构应该"尖叫"出其用途——看到源码应该知道"这是一个医疗系统"，而不是"这用了 Spring"。

- 架构以用例为中心，框架只是工具
- 应能在不依赖任何框架的情况下进行单元测试

### 服务不等于架构

- 服务化的两大谬论：解耦合谬论（共享数据仍然耦合）、独立部署谬论（行为耦合仍需协调）
- 架构边界不落在服务之间，而是**穿透服务**，在服务内部以组件形式存在
- 横跨型变更（cross-cutting concern）是按功能划分服务最大的敌人

### 测试边界

- 测试是系统的一部分，位于架构最外圈，依赖方向指向内层
- **脆弱测试问题**：通过 GUI 验证系统的测试必然脆弱
- 为测试创建专用 API，业务逻辑应不通过 GUI 也能被测试
- 不要让每个产品类对应一个测试类——测试 API 应隐藏应用结构

## 9. 代码组织方式

四种方式，能力递增：

| 方式 | 特点 | 局限 |
|------|------|------|
| **按层封装** | 水平分层（Web/Service/Repository） | 无法展现业务领域，容易绕过层次 |
| **按功能封装** | 垂直切分，相关类放同一包 | 顶层展现业务，但隔离不够 |
| **端口和适配器** | 区分 Domain 和 Infrastructure | 需要访问修饰符配合 |
| **按组件封装** | 粗粒度组件对外提供干净接口 | 最接近微服务的准备状态 |

**关键**：如果所有类型都设为 public，这四种方式在语法上完全没有区别。**利用编译器（访问修饰符）执法**，让架构原则在编译期可强制执行。

## 10. 实战决策指南

### 划分边界的时机

- **边界画在不相关的事情中间**：GUI 与业务逻辑、数据库与业务逻辑
- **边界沿变更轴来画**：两侧以不同原因、不同速率变化
- YAGNI 与预见性设计之间需要平衡
- **这不是一次性决定**——持续观察系统演进，在拐点出手

### 判断是否需要边界

观察信号：
- 两个模块的变更原因是否总是不同？
- 修改一处是否经常波及另一处？
- 团队的不同小组是否在同一个模块上冲突？

如果以上问题的答案为"是"，就需要边界。

### FitNesse 案例的启示

- 自建 Web 服务器延后了框架决策
- 在业务逻辑与数据存储间加 `WikiPage` 接口，先用内存实现
- **18 个月不用数据库**——不需要面对表结构、查询、密码
- 客户需要时，一天就写出了 MySQL 实现
- 结论：好的边界让"推迟决策"变为现实

### 过早决策的反面教训

- **P 公司**：开发初期采用三层富架构，每个简单字段需修改三层类、四个消息协议、八个处理函数。从未部署过需要集群的系统。
- **W 公司**：全套企业级 SOA 管理一个小型车队业务。添加一个联系人需要经过 ServiceRegistry、ContactService 等复杂流程。

### 视频销售网站案例

完整设计流程示范：
1. 识别角色（观看者、购买者、管理员、视频作者）——SRP 驱动
2. 按用例划分——每个角色的用例独立
3. 组件架构——视图、展示器、交互器、控制器分层，再按角色分组
4. 依赖管理——"使用"关系与控制流同向，"继承"关系与控制流反向（OCP 体现）
5. 灵活部署——可合并为 5 个或 2 个 jar，按需调整

**两个维度的隔离**：按 SRP 隔离角色（变更原因不同），按依赖规则隔离层级（变更速率不同）。
